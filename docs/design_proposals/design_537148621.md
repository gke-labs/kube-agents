# Technical Design Proposal: Agent Pod RBAC & GCP IAM Scope Reduction

- **Task ID:** Buganizer Task 537148621
- **Title:** Agent Pod RBAC & GCP IAM Scope Reduction
- **Target Files:**
  - `k8s-operator/internal/controller/platformagent_manifests.go`
  - `k8s-operator/internal/controller/platformagent_controller.go`
  - `k8s-operator/config/agent_rbac/platformagent.yaml`
  - `k8s-operator/scripts/provision_03_gcp_iam.sh`
  - `k8s-operator/scripts/common.sh`
  - `agents/platform/SOUL.md`
- **Findings Addressed:** LP-001, LP-002, LP-003, THREAT-005, MCP-007

---

## 1. Executive Summary & Problem Statement

Currently, the Platform Agent in `kube-agents` operates under default RBAC and IAM configurations that grant excess privileges beyond what is strictly necessary for its fleet-monitoring and GitOps workflow responsibilities:

1. **Kubernetes RBAC Over-Privileging (LP-001, LP-002):**
   - The operator controller currently binds the Platform Agent's ServiceAccount to the cluster-wide standard `view` ClusterRole via a `ClusterRoleBinding` (`kubeagents:viewer:<ns>:<name>`).
   - The operator also reconciles a custom `explorer` ClusterRole (`kubeagents:explorer:<ns>:<name>`) that grants `get` and `list` on `nodes`, `pods`, `namespaces`, and `customresourcedefinitions`.
   - Granting broad `view` privileges across all namespaces, coupled with unconstrained cluster-wide object listing, creates an excessively large security exposure if an agent container is compromised.

2. **GCP IAM Over-Privileging (LP-003, THREAT-005):**
   - `k8s-operator/scripts/provision_03_gcp_iam.sh` defaults `PLATFORM_AGENT_PERMISSION_SET` to `gke-admin`, which grants `roles/container.clusterAdmin`, `roles/container.admin`, and `roles/monitoring.admin` to the agent's GCP Service Account.
   - This directly contradicts the security guarantees documented in `agents/platform/SOUL.md`, which states that the Platform Agent operates strictly with read-only cluster visibility and relies on declarative GitOps (Pull Requests) for infrastructure modifications.

3. **Missing Static Declarative RBAC Configuration (MCP-007):**
   - No static declarative RBAC manifest exists at `k8s-operator/config/agent_rbac/platformagent.yaml`. Non-operator or static Helm/Kustomize deployments lack a canonical reference for least-privilege agent RBAC.

## 2. Current State & Code Comments Analysis

### 2.1 Codebase Inspection

- **`k8s-operator/internal/controller/platformagent_manifests.go` & `platformagent_controller.go`**:
  - `reconcileRBAC` creates a `ClusterRoleBinding` for `view` and builds `buildPlatformExplorerRole` for `nodes`, `pods`, `namespaces`, `customresourcedefinitions`.
  - `handleDeletion` cleans up `viewerBindingName`, `explorerBindingName`, and `explorerRoleName`.
- **`k8s-operator/scripts/provision_03_gcp_iam.sh` & `common.sh`**:
  - `get_platform_agent_roles()` defaults to `gke-admin` via `${PLATFORM_AGENT_PERMISSION_SET:-gke-admin}`.
  - `common.sh` line 163 prompts with `"gke-admin"` as the default permission set.
  - Code comment in `provision_03_gcp_iam.sh` line 170: `# The agent can query logs for diagnostics but must not administer the audit-log sink.`
- **`agents/platform/SOUL.md`**:
  - Core Truths (Least Privilege Constraint):
    > You operate with standard GKE Read-Only cluster visibility for fleet auditing, and hold highly restricted, elevated namespace write permissions exclusively for the specific Custom Resources (CRs) that declare and manage your agent identity. You do not hold general write permissions for other infrastructure workloads.

### 2.2 Potential Code Comment & Architectural Conflicts / Questions Raised

1. **Question on Backward-Compatible Deletion Cleanup:**
   - *Observation:* Code comment in `platformagent_controller.go` line 154 (`// Delete Viewer ClusterRoleBinding`) and line 160 (`// Delete Explorer ClusterRoleBinding`) highlights that finalizer deletion explicitly deletes legacy RBAC objects by name (`kubeagents:viewer:<ns>:<name>` and `kubeagents:explorer:<ns>:<name>`).
   - *Question for Proposal:* Should the finalizer cleanup logic be preserved to handle existing clusters upgrading to the new reduced RBAC model?
   - *Design Decision:* Yes. `handleDeletion` will attempt to delete both legacy names (`kubeagents:viewer:...`, `kubeagents:explorer:...`) and new scoped names to ensure clean teardown without orphan cluster role bindings during upgrades.

2. **Question on CRD and KCC Scope:**
   - *Observation:* Finding LP-001/LP-002 requires removing CRUD on KCC (Config Connector) clusters/CRDs. `buildPlatformExplorerRole` currently grants `get` and `list` on `customresourcedefinitions`.
   - *Question for Proposal:* Does removing write/CRUD access to CRDs or Config Connector resources impair agent execution?
   - *Design Decision:* No. `SOUL.md` explicitly forbids manual API mutation of infrastructure resources ("All GKE infrastructure changes ... must be automated via the active declarative workflow"). Read-only (`get`, `list`) access to CRD metadata remains permitted for fleet auditing, but no `create`, `update`, `patch`, or `delete` verbs will be granted on CRDs or KCC CRs.

## 3. Proposed Technical Design

```
+-----------------------------------------------------------------------------------+
|                                Platform Agent Pod                                 |
+-----------------------------------------------------------------------------------+
                                   |                     |
             Workload Identity     |                     | K8s ServiceAccount
                                   v                     v
+------------------------------------+   +------------------------------------------+
|          GCP IAM Scope             |   |            K8s RBAC Scope                |
|       (roles/read-only)            |   |   (Minimal ClusterRole + Local Role)     |
+------------------------------------+   +------------------------------------------+
| - roles/container.clusterViewer    |   | ClusterRole (fleet-audit):               |
| - roles/container.viewer           |   | - nodes, namespaces, pods, services (get/|
| - roles/monitoring.viewer          |   |   list/watch)                            |
| - roles/logging.viewer             |   | - deployments, statefulsets (get/list)  |
| - roles/iam.securityReviewer       |   | - customresourcedefinitions (get/list)   |
| - roles/mcp.toolUser               |   |                                          |
| (NO container.clusterAdmin)        |   | Role (namespace-local):                  |
|                                    |   | - platformagents CR (get/list/watch/     |
|                                    |   |   update/patch status)                   |
+------------------------------------+   +------------------------------------------+
```

### 3.1 Kubernetes RBAC Scope Reduction
1. **Replace `view` ClusterRoleBinding & Refine ClusterRole:**
   - Remove the binding to the broad Kubernetes built-in `view` ClusterRole.
   - Replace `buildPlatformExplorerRole` with `buildMinimalPlatformRole`:
     - Grants `get`, `list`, `watch` on core audit resources: `nodes`, `namespaces`, `pods`, `services`, `events`, `persistentvolumes`.
     - Grants `get`, `list`, `watch` on apps resources: `deployments`, `statefulsets`, `daemonsets`.
     - Grants `get`, `list` on `apiextensions.k8s.io/customresourcedefinitions`.
     - Strictly excludes `secrets` and prevents any write/mutation capabilities on cluster-scoped resources or KCC CRDs.
2. **Namespace-Scoped Role for Agent CRs:**
   - Create a local `Role` in the agent's deployment namespace (`agent.Namespace`) for managing its own `PlatformAgent` Custom Resources (`kubeagents.x-k8s.io`).

### 3.2 GCP IAM Scope Reduction
1. **Update `provision_03_gcp_iam.sh` Default:**
   - Change fallback from `gke-admin` to `read-only`:
     ```bash
     PLATFORM_AGENT_PERMISSION_SET="${PLATFORM_AGENT_PERMISSION_SET:-read-only}"
     ```
2. **Update `common.sh` Default:**
   - Update default prompt value for `PLATFORM_AGENT_PERMISSION_SET` from `"gke-admin"` to `"read-only"`.

### 3.3 Static Declarative RBAC (`k8s-operator/config/agent_rbac/platformagent.yaml`)
Create `k8s-operator/config/agent_rbac/platformagent.yaml` with static YAML manifests defining:
- `ServiceAccount`: `kubeagents-platform-agent` in `kubeagents-system`.
- `ClusterRole`: `kubeagents-platform-agent-reader` with audit-only read permissions.
- `ClusterRoleBinding`: `kubeagents-platform-agent-reader-binding`.
- `Role`: `kubeagents-platform-agent-local` in `kubeagents-system` for `platformagents` CRs.
- `RoleBinding`: `kubeagents-platform-agent-local-binding`.

### 3.4 `SOUL.md` Alignment
- Ensure `agents/platform/SOUL.md` accurately describes the reduced RBAC and default `read-only` IAM posture.
